💡 What: Changed the `agent_run` persistence flow to execute a single UPSERT (via POST to PostgREST with `on_conflict` specified) instead of conditionally executing a SELECT query and then performing a PATCH or UPSERT.
🎯 Why: Previously, persisting repo memory fallback resulted in multiple sequential network requests because it executed a SELECT query followed conditionally by a PATCH. By using UPSERT directly with `on_conflict=thread_id,input_hash`, we avoid an unnecessary network round-trip.
📊 Impact: Reduced the fallback path network requests from 6 down to 5, resulting in roughly a 16.6% reduction in latency for this particular non-RPC path.
🔬 Measurement: Created a micro-benchmark using a mocked global fetch. It verified that the total number of fetch requests issued on fallback drops exactly by one per loop iteration (from 600 to 500 across 100 turns, shaving simulated network times by 16.6%).
