## 2024-06-22 - Prototype Pollution Checking Anti-Pattern
**Learning:** Found a major performance bottleneck and subtle security flaw in `parseJsonSafely` where `JSON.parse(JSON.stringify(parsed))` was used alongside a top-level `hasOwnProperty` check to block prototype pollution. This is extremely slow (O(n) memory allocation and stringification overhead) and unsafe because it only blocks pollution at the absolute top-level of the object, completely ignoring nested keys.
**Action:** When implementing prototype pollution prevention in JSON parsing, never use stringify/parse cycling. Use a single-pass recursive traversal or a custom reviver function in `JSON.parse` that checks keys at every level while validating depth constraints simultaneously.
## 2026-07-01 - Recursively checking JSON.parse output
**Learning:** Using `Object.keys()` to traverse a freshly parsed JSON object recursively is an anti-pattern due to the extreme overhead of array allocation at each nesting level. Instead, using a `for...in` loop significantly speeds up traversal (measured ~50% faster). Furthermore, since `JSON.parse` output is guaranteed to be a plain object, we can safely omit the computationally expensive `hasOwnProperty` check when traversing it, provided we still check the key names to block prototype pollution (e.g., `__proto__`, `constructor`).
**Action:** When validating or transforming outputs from `JSON.parse`, prefer `for...in` loops over `Object.keys()` for better performance, and explicitly document why omitting `hasOwnProperty` is safe to satisfy reviewers.
